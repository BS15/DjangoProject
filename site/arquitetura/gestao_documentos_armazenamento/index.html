Gestão de Documentos e Armazenamento

Este documento descreve como o PaGé gerencia anexos/documentos no backend: validação, persistência no storage, convenção de pastas, acesso seguro e comportamento no sistema de arquivos.

1. Visão geral técnica

No Django, os anexos são armazenados em FileField e o banco salva apenas o caminho relativo do arquivo (ex.: pagamentos/2026/proc_123/nota.pdf).

  1. Upload via formulário (multipart/form-data) em action (POST).
  2. Validação por MIME real em validar_arquivo_seguro.
  3. Definição de caminho com upload_to (majoritariamente caminho_documento).
  4. Persistência no storage + caminho relativo no banco.
  5. Leitura/download preferencial por endpoint seguro.

2. Onde os arquivos ficam fisicamente

  • MEDIA_URL = /media/
  • MEDIA_ROOT = BASE_DIR/media

Em dev local: ./media/<caminho_relativo>.

Em Docker Compose: arquivos em /app/media/ (volume media_volume).

3. Convenção de pastas

3.1 Helper canônico: caminho_documento

  • Processo: pagamentos/<ano>/proc_<id>/<arquivo>
  • Prestação diária: verbasindenizatorias/prestacoes/prestacao_<id>/<arquivo>
  • Diária/Reembolso/Jeton/Auxílio: subpastas dedicadas em verbasindenizatorias/
  • Suprimentos: suprimentosdefundos/suprimento_<id>/...
  • Fallback: documentos_avulsos/<arquivo>

3.2 Pastas especiais

  • processos_arquivados/ (consolidado final)
  • devolucoes/
  • assinaturas_rascunho/ e documentos_assinados/
  • fiscal/pagamentos_impostos/<YYYY-MM>/
  • temp/ (arquivos temporários)

4. Higienização de nomes

O helper _safe_filename remove diretórios embutidos, normaliza o nome e evita path traversal.

5. Validação de upload

validar_arquivo_seguro aceita apenas application/pdf, image/jpeg e image/png, com detecção por conteúdo binário real.

6. Acesso seguro aos arquivos

Endpoint: /documentos/secure/<tipo_documento>/<documento_id>/ via download_arquivo_seguro.

A view resolve o documento, valida autorização contextual, registra acesso em RegistroAcessoArquivoProcessual e retorna FileResponse.

7. Exposição de /media

O Nginx expõe /media/ com alias para /app/media/. Isso atende URLs diretas de arquivo.url, mas bypassa checagens contextuais da rota segura.

8. Limpeza de arquivos

Para todos os documentos com FileField no sistema, existe limpeza explícita de arquivo anterior/físico via _delete_file em sinais (pre_save, post_delete).

8.1 Modelos cobertos

  • Documentos de processo: DocumentoProcesso, DocumentoOrcamentarioProcessual, ComprovantePagamento
  • Processo consolidado: Processo.arquivo_final
  • Devoluções e suporte: DevolucaoProcessual.comprovante, AssinaturaEletronica (rascunho e assinado)
  • Documentos de verbas: DocumentoDiaria, DocumentoComprovacao, DocumentoReembolso, DocumentoJeton, DocumentoAuxilio
  • Documentos de suprimento: DocumentoSuprimentoDeFundos, DespesaSuprimento, PrestacaoContasSuprimento.comprovante_devolucao
  • Documentos fiscais: DocumentoPagamentoImposto (3 campos)

8.2 Padronização completa

A estratégia de cleanup automático é uniforme. Cada modelo com FileField implementa:

  1. @receiver(post_delete) para remover arquivo ao deletar registro
  2. @receiver(pre_save) para remover versão anterior se arquivo for sobrescrito

Não há exceções: todos os agregados documentais seguem o mesmo padrão.

9. Compatibilidade com storage remoto

Serviços documentais usam storage.exists e arquivo.open("rb"), sem depender de caminho local, o que mantém compatibilidade com storages cloud.

10. Resumo operacional

  • Banco guarda caminho relativo; arquivo vive em MEDIA_ROOT/volume.
  • Conveção de pastas centralizada em caminho_documento.
  • Existe rota de download seguro com auditoria de acesso.
  • Há exposição HTTP de /media/ para links diretos.
  • Upload é validado por MIME real (PDF/JPEG/PNG).
